I've just released [BlogMore](https://blogmore.davep.dev/)
[v2.37.0](https://blogmore.davep.dev/changelog/#v2370). This is a small
update that fixes a bug and also improves the way that warnings and errors
are emitted.

The bugfix comes from [yesterday's
post](/2026/06/02/recreating-my-blog-stats.html) where I noticed that the
[stats page](/stats/) was misreporting the number of tags on the blog. The
cause of the problem was that, within reason, a single tag can be
represented in a number of different ways. In the `tags` frontmatter you
could write any of `Python Coding`, `Python coding`, `python coding`,
`Python-Coding`, etc and they would all be seen as the tag `python-coding`.
This gives some useful flexibility and also ensures that the actual tag is
URL-safe.

The problem was that the stats page was counting the list of unique tags, as
they were typed into posts, rather than the unique list of *"made safe"*
tags. So [that is now fixed, along with categories
too](https://github.com/davep/blogmore/pull/582/changes).

The other main change in this release is that warnings and errors that might
get printed during post parsing and other operations are now printed to
[`stderr`](https://en.wikipedia.org/wiki/Standard_streams#Standard_error_(stderr)).
Until now everything was simply printed to
[`stdout`](https://en.wikipedia.org/wiki/Standard_streams#Standard_output_(stdout)),
which was fine until I introduced the [`links
dump`](https://blogmore.davep.dev/command_line/#dump) and [main
`dump`](https://blogmore.davep.dev/command_line/#dump-command)
[commands](https://blogmore.davep.dev/command_line/). With the addition of
those commands, you're far more likely to want to redirect the output to a
file. So if there's some sort of warning when parsing a post, *that* would
get dropped into the middle of the JSON output (if using `dump`), resulting
in a JSON file that can't be parsed.

Now all warnings and errors will go to `stderr`.
