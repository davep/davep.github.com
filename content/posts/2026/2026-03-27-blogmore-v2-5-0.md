I've released [BlogMore](https://blogmore.davep.dev/) v2.5.0 out into the
world. This release is the result of [an
observation](https://mastodon.me.uk/@andyc/116290194989779553)
[Andy](https://mastodon.me.uk/@andyc) made about [the Markdown library used
in BlogMore](https://python-markdown.github.io) (it might apply to
[MarkdownIT](https://github.com/executablebooks/markdown-it-py) too, which
would of course affect [Hike](https://hike.davep.dev/)): it doesn't support
strikethrough markup out of the box.

I'm not sure I've ever used that markup anywhere on my blog, but I've used
it often enough on GitHub (for example) that I just assume it's going to be
there (and now I think about it Hike might be okay 'cos it uses
[Textual](https://textual.textualize.io)'s [Markdown
widget](https://textual.textualize.io/widgets/markdown/) which [I know for
certain](https://github.com/Textualize/textual/graphs/contributors) [uses
GitHub-flavoured
Markdown](https://github.com/Textualize/textual/blob/e84eacc1899f940ea068c4ae5c7eca2dd4a489db/src/textual/widgets/_markdown.py#L1313-L1317)).
But, for sure, the Markdown library doesn't implement it because it's [not
part of the original approach to
Markdown](https://python-markdown.github.io/#goals).

On the other hand: implementing a form of strikethrough markup [is one of
the samples in the documentation on writing
extensions](https://python-markdown.github.io/extensions/api/#example_3), so
it seemed like a very reasonable thing to add. Given this, [one quick prompt
to the agents later](https://github.com/davep/blogmore/issues/345) and
[strikethrough was added](https://github.com/davep/blogmore/pull/346).

Now I'm going to ~~totally~~ have to find a reason to use this markup from
time to time ~~to time~~.
